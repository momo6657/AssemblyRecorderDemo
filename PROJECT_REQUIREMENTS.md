## 🎯 项目核心需求

### 1. 功能需求

#### 1.1 手机端功能（必需）
- ✅ **模型导入**
  - 支持 GLB/GLTF 格式
  - 本地文件选择器
  - 自动材质转换（URP Lit）
  - 自动添加 Collider
  - 状态：已完成

- ✅ **任务创建**
  - 上传模型到服务器
  - 创建任务元数据
  - 任务状态管理（pending/in_progress/completed）
  - 状态：已完成

- ✅ **录制列表**
  - 显示所有录制（不按任务过滤）
  - 支持点击加载录制
  - 显示录制标题和时间
  - 状态：已完成

- ✅ **步骤回放**
  - 加载录制步骤数据
  - 轨迹插值动画播放
  - 步骤列表显示
  - Next/Prev 导航
  - 模型在当前位置播放（无偏移）
  - 状态：已完成

#### 1.2 Quest 端功能（必需）
- ✅ **任务轮询**
  - 自动轮询服务器获取 pending 任务
  - 下载模型文件
  - 加载模型到场景
  - 状态：已完成

- ✅ **装配录制**
  - 手柄抓取零件
  - 记录轨迹（position/rotation/scale）
  - 采样频率可配置
  - 状态：已完成

- ⚠️ **录制上传**
  - 创建 recording 元数据
  - 上传步骤数据
  - 状态：已完成，但有 Bug（抓握时模型缩小）

#### 1.3 服务端功能（必需）
- ✅ **任务管理**
  - POST /createTask
  - GET /pollTask
  - 状态：已完成

- ✅ **模型管理**
  - POST /uploadModel
  - GET /downloadModel
  - 状态：已完成

- ✅ **录制管理**
  - POST /createRecording
  - POST /uploadRecordingSteps
  - GET /listRecordings
  - GET /downloadRecording
  - GET /downloadRecordingSteps
  - 状态：已完成

### 2. 非功能需求

#### 2.1 性能要求
- ✅ 模型加载时间：< 2 秒（19MB GLB）
- ✅ 步骤回放流畅度：60 FPS
- ✅ 网络请求超时：30 秒
- ⚠️ 避免重复加载模型（已优化，添加缓存机制）

#### 2.2 用户体验要求
- ✅ 动画播放平滑（轨迹插值）
- ✅ 模型在用户面前播放（无位置跳动）
- ✅ 操作响应及时（< 100ms）
- ✅ 错误提示清晰

#### 2.3 稳定性要求
- ✅ 网络异常处理
- ✅ 模型加载失败处理
- ⚠️ 避免破坏已修复的 Bug
- ⚠️ 日志完整可追溯

## 📝 需求变更流程

### 如何修改需求

#### 1. 用户提出需求变更
用户可以随时提出需求变更，例如：
- "我想添加 XXX 功能"
- "我不需要 XXX 功能了"
- "XXX 功能的优先级改为 P0"

#### 2. Agent 更新本文档
任何 Agent 都可以根据用户指示修改本文档：

**步骤 A：更新版本信息**
```markdown
### 当前版本
- **版本号**：v1.1.0  ← 更新版本号
- **最后更新**：2026-03-05  ← 更新日期
- **更新者**：AI Agent  ← 记录更新者
- **变更说明**：添加 XXX 功能需求  ← 说明变更内容
```

**步骤 B：更新版本历史表**
```markdown
| v1.1.0 | 2026-03-05 | AI Agent | 添加 XXX 功能需求 |
```

**步骤 C：更新相应章节**
- 如果是新增功能：添加到"功能需求"章节
- 如果是删除功能：移动到"明确不做的功能"章节
- 如果是优先级调整：更新"已知问题清单"
- 如果是目标调整：更新"当前阶段目标"

**步骤 D：同步更新其他文档**
- 更新 `PROJECT_README.md` 的"项目变更记录"
- 在 `PROJECT_CHANGELOG.md` 中添加条目
- 如果影响架构，更新 `PROJECT_README.md` 的"技术架构"章节

#### 3. 通知用户
Agent 完成更新后，向用户确认：
```
✅ 已更新需求文档：
- 版本：v1.0.0 → v1.1.0
- 变更：添加 XXX 功能需求
- 影响文档：PROJECT_REQUIREMENTS.md, PROJECT_README.md, PROJECT_CHANGELOG.md
```

## 🔒 需求约束条件

### 技术约束
- ✅ 必须使用 Unity 2022+
- ✅ 必须支持 Android 平台
- ✅ 必须支持 Meta Quest (OpenXR)
- ✅ 必须使用 glTFast 加载 GLB/GLTF
- ✅ 必须使用 URP 渲染管线

### 业务约束
- ✅ 手机端和 Quest 端必须通过服务器同步
- ✅ 录制数据必须持久化
- ✅ 支持多个录制（不限制数量）
- ⚠️ 暂不支持多用户隔离（userId 预留但不强制）

### 资源约束
- ✅ 开发团队：1 人 + AI Agent
- ✅ 测试设备：Android 手机 + Meta Quest
- ✅ 服务器：本地开发服务器（Flask）
- ✅ 预算：Demo 阶段，无预算限制

## 📊 需求优先级定义

### P0 - 阻塞性（Critical）
- 影响核心功能无法使用
- 导致数据丢失或损坏
- 严重影响用户体验
- **必须立即修复**

### P1 - 重要（High）
- 影响重要功能
- 影响部分用户体验
- 有临时解决方案
- **尽快修复**

### P2 - 一般（Medium）
- 影响次要功能
- 用户体验轻微影响
- 有替代方案
- **可延后修复**

### P3 - 优化（Low）
- 性能优化
- 代码重构
- 文档完善
- **有时间再做**
